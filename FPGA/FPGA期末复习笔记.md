# 第1章 数字系统设计与Verilog HDL 


## 1. 数字系统设计技术

### 1.1 Top-down 设计（自顶向下）

- **流程**：从系统级行为描述开始，逐步分解为子模块，直至门级/物理实现。
- **特点**：反复进行“设计 → 验证 → 修改 → 再验证”，直到满足功能、速度、功耗、价格和可靠性的平衡。
- **优势**：适合复杂系统，便于早期仿真和错误发现。

### 1.2 Bottom-up 设计（自底向上）

- **流程**：从基础元件（门、触发器、计数器等）开始，组合成更大的功能模块。
- **缺点**：效率低，易出错，不适合大规模复杂设计。

### 1.3 IP 复用技术与 SoC

| 术语 | 全称 / 含义 |
|------|-------------|
| **IP核** | Intellectual Property Core – 功能完整、已验证、可重用的电路功能模块 |
| **软IP** | 以HDL描述，不涉及具体电路实现 |
| **固IP** | 已完成综合的功能块 |
| **硬IP** | 最终阶段产品（如掩膜） |
| **IP复用** | 在不同设计中重复使用已验证的IP核，提高开发效率 |
| **SoC** | System on a Chip – 单片系统，将整个系统集成在一个芯片上 |

## 2. 数字系统设计流程

```mermaid
flowchart LR
    A[设计输入] --> B[综合]
    B --> C[适配/结构综合]
    C --> D[仿真]
    D --> E[编程/配置]
```

### 各步骤详解

| 步骤 | 说明 | 产物/工具 |
|------|------|-----------|
| **设计输入** | 原理图或HDL文本（Verilog/VHDL等） | 原理图编辑器、HDL编辑器 |
| **综合** | 将高层描述（行为/RTL）转化为门级网表<br>(即能将原理图或HDL语言描述的电路功能转化为具体电路结构网表的工具) | 综合器（Synplify, FPGA Compiler等） |
| **适配（结构综合）** | 将网表适配到指定目标器件，生成下载文件 | 适配器（对CPLD生成JEDEC，对FPGA生成Bitstream） |
| **仿真** | 功能仿真（前仿真）验证逻辑功能；时序仿真（后仿真）验证时序 | ModelSim, NC-Sim等 |
| **编程/配置** | 将编程文件装入PLD器件 | 编程器/下载电缆（EEPROM工艺称编程，SRAM工艺称配置） |

## 4. 几个专有名词解释

### 4.1 基本术语对照表

| 缩写 | 全称 | 定位 / 特点 |
|------|------|-------------|
| **CPU** | Central Processing Unit | 通用全能“大脑”，串行强，GHz级，高功耗 |
| **MCU** | Microcontroller Unit | 集成CPU+RAM+Flash+外设的小电脑，低功耗，MHz级，跑裸机/RTOS |
| **GPU** | Graphics Processing Unit | 并行计算大力士，超多小核，适合渲染/AI，高功耗 |
| **TPU** | Tensor Processing Unit | 专为机器学习/深度学习设计的张量处理器，AI能效比高 |
| **NPU** | Neural Processing Unit | 神经网络处理器（常用于移动端AI加速） |
| **DSP** | Digital Signal Processor | 专业算力员，擅长高速数字信号处理（FFT、滤波、电机控制等） |

### 4.2 详细对比表

#### CPU vs MCU vs GPU

| 对比项 | CPU | MCU | GPU |
|--------|-----|-----|-----|
| 核心定位 | 通用全能大脑 | 单片小电脑 | 并行计算大力士 |
| 核心特点 | 单核/多核，串行强 | 集成度高，低功耗 | 超多小核，并行极强 |
| 速度 | 极快（GHz级） | 较慢（MHz级） | 算力极高，并行快 |
| 组成 | 仅运算核心，需外接存储 | 片内CPU+RAM+Flash+外设 | 专注图形/并行，需配合CPU |
| 功耗 | 较高（W~百W） | 极低（mW~W） | 高（W~百W） |
| 运行系统 | Windows/Linux/Android | 裸机、简易RTOS | 不跑主系统，受CPU调度 |
| 典型应用 | 电脑、手机、服务器 | 家电、Arduino、STM32、工控 | 游戏、渲染、AI、视频处理 |

#### GPU vs TPU

| 对比项 | GPU | TPU |
|--------|-----|-----|
| 设计初衷 | 图形渲染、游戏 | 机器学习/深度学习 |
| 核心擅长 | 通用并行计算 | 矩阵运算、张量运算（AI专属） |
| 通用性 | 极强（能玩游戏、渲染、AI） | 极弱（基本只干AI） |
| 能效比 | 不错 | 比GPU高很多（同功耗AI更快） |
| 代表厂商 | NVIDIA、AMD、ARM | Google |
| 使用场景 | 个人电脑、显卡、云端训练 | 谷歌云端AI、搜索、翻译、TPU集群 |

#### MCU vs DSP

| 场景 | MCU | DSP |
|------|-----|-----|
| 信号处理复杂度 | 中等（简单音频、传感器） | 极高（FFT、滤波、雷达、电机FOC） |
| 实时性要求 | 较高（毫秒级） | 极高（微秒级） |
| 算力需求 | 中 | 高（MAC密集） |
| 成本 | 有限 | 充足 |
| 开发难度 | 较低（通用MCU生态） | 较高（专用工具/架构） |
| 定位 | 全能小管家（控制和调度） | 专业算力员（复杂高速计算） |

#### MCU vs FPGA

| 对比项 | MCU | FPGA |
|--------|-----|------|
| 运算方式 | 串行，逐条执行指令 | 全并行，多任务同时跑 |
| 实时性 | 一般，受中断/调度影响 | 极强，硬件级低延迟 |
| 灵活性 | 改程序即可 | 改硬件电路结构 |
| 开发难度 | 低，C语言为主 | 高，需硬件描述语言（Verilog/VHDL） |
| 功耗 | 低，适合长期待机 | 较高，静态功耗偏大 |
| 成本 | 便宜，几元~几十元 | 贵，几十元~上千元 |
| 适用场景 | 控制、交互、简单计算、物联网 | 高速接口、信号处理、协议转换、算法加速 |

> **总结**：MCU = 便宜、简单、好开发，适合“控制为主”；FPGA = 并行、高速、强实时，适合“性能/硬件定制”。

# 第2章 FPGA/CPLD 器件



## 2.1 PLD 器件概述

- **PLD**（Programmable Logic Device，可编程逻辑器件）：用户可配置逻辑功能的数字集成电路。
- **发展历程**：PROM → PLA → PAL → GAL → CPLD → FPGA。
- **里程碑**：
  - 1985年，Xilinx公司推出首款**FPGA**（Field Programmable Gate Array，现场可编程门阵列）。
  - **CPLD**（Complex Programmable Logic Device，复杂可编程逻辑器件）由EPLD改进而来。
- **分水岭**：GAL22V10（约500~750门）作为简单PLD（SPLD）与高密度PLD的分界。

### SPLD 四种器件对比（与阵列/或阵列/输出电路可编程性）

| 器件 | 与阵列 | 或阵列 | 输出电路 |
|------|--------|--------|----------|
| PROM | 固定 | 可编程 | 固定 |
| PLA  | 可编程 | 可编程 | 固定 |
| PAL  | 可编程 | 固定   | 固定 |
| GAL  | 可编程 | 固定   | 可组态（OLMC） |

### 按编程次数分类
- **OTP**（One Time Programmable，一次编程）
- **多次可编程**（科研开发常用）

### 按编程元件/工艺分类
| 类型 | 工艺 | 典型器件 | 是否易失 |
|------|------|----------|----------|
| 熔丝（Fuse） | 烧断连接 | 早期PROM | 非易失 |
| 反熔丝（Antifuse） | 高压击穿形成连接 | 某些军用FPGA | 非易失 |
| EPROM | 紫外线擦除，电编程 | 早期PLD | 非易失 |
| EEPROM | 电擦除电编程 | 多数CPLD | 非易失 |
| Flash | 快闪存储 | 部分CPLD | 非易失 |
| **SRAM** | 静态存储器 | **多数FPGA** | **易失**（需外挂配置芯片） |

## 2.2 PLD 的基本原理与结构

### 通用结构
所有PLD均由**与阵列**、**或阵列**和**输出电路**三部分组成。

- **与阵列**：产生乘积项（最小项）
- **或阵列**：将乘积项相或，实现逻辑函数
- **输出电路**：可组合/时序输出（如寄存器、MUX等）

### PLD器件的结构原理图
```mermaid
flowchart LR
    A[输入信号] --> B[输入缓冲电路]
    B --> C[与阵列<br>（产生乘积项）]
    C --> D[或阵列<br>（乘积项相或）]
    D --> E[输出缓冲电路]
    E --> F[输出信号]
    E --> B
```

### PLD阵列图中的连接表示
- **固定连接**：实心黑点（●）
- **可编程连接**：叉号（×）
- **未连接**：无标记

### 输入缓冲电路
- 提供原变量和反变量（互补输出），用于与阵列输入。

## 2.3 低密度 PLD 的原理与结构

### 2.3.1 PROM（Programmable Read-Only Memory，可编程只读存储器）
- **结构**：与阵列**固定**（全译码），或阵列**可编程**。即本质是一个**与或阵列**。
- **特点**：实现任意组合逻辑，但芯片面积大，仅适合小规模。


### 2.3.2 PLA（Programmable Logic Array，可编程逻辑阵列）
- **结构**：与阵列**可编程**，或阵列**可编程**。即本质是一个**与或阵列**。
- **优点**：更灵活，可减少乘积项数量，节省面积。
- **缺点**：综合工具复杂。

### 2.3.3 PAL（Programmable Array Logic，可编程阵列逻辑）
- **结构**：与阵列**可编程**，或阵列**固定**。即本质是一个**与或阵列**。
- **特点**：输出逻辑固定，速度快，但灵活性稍差。

### 2.3.4 GAL（Generic Array Logic，通用阵列逻辑）
- **结构**：与阵列可编程，或阵列固定，但**输出电路可组态**。即本质是一个**与或阵列**。
- **核心**：**OLMC**（Output Logic Macro Cell，输出逻辑宏单元）。
- **OLMC特点**：
  - 可配置为组合输出（直接输出或经异或）或时序输出（带寄存器）。
  - 通过编程选择MUX（多路选择器）控制反馈路径、时钟/使能等。
- **典型器件**：GAL22V10（含10个OLMC，每个可独立配置）。

**PROM、PLA、PAL、GAL之间的核心区别，不在于“是不是”与或阵列，而在于“与阵列”和“或阵列”谁是可以编程修改的，以及输出电路是否固定。**

## 2.4 CPLD 的原理与结构

### CPLD 总体结构
- 由多个**逻辑块（LAB/GLB）**（或称为**宏单元**）通过**全局互连总线(可以统一理解为PIA)** 连接。
- 每个宏单元包含：乘积项阵列（与阵列）、或门、可编程寄存器、MUX等。

### 典型CPLD结构示例
- **Altera MAX 7000S**：
  - 逻辑阵列块（LAB, Logic Array Block）包含16个宏单元。
  - 宏单元含可编程与阵列（乘积项）、或门、触发器、MUX。
  - 全局布线（PIA，Programmable Interconnect Array）连接各LAB。

- **Lattice ispLSI 1032**：
  - GLB（Generic Logic Block，通用逻辑块）：含与阵列、或门、寄存器。
  - GRP（Global Routing Pool，全局布线池）实现互连。

- **Xilinx XC9500**：
  - 每个宏单元含18个乘积项（其中3个可分配），带可编程寄存器。

### CPLD 特点
- **非易失**（EEPROM/Flash工艺），掉电不丢失。
- **乘积项结构**（与-或阵列），适合复杂组合逻辑和有限状态机。
- 延时**可预测**，但容量较小。

## 2.5 FPGA 的原理与结构

### 核心原理：查找表（LUT，Look-Up Table）
- **LUT** 本质上是一个小容量RAM，存储所有输入组合对应的输出值。
- 例如：2输入LUT可实现任意2输入逻辑函数（存储4位真值表）。
- **示例**：3人表决电路（输入A,B,C，输出F=多数为1）：
  - 真值表列出8种组合，将对应输出存入3输入LUT的8个存储单元。
  - 输入作为地址，输出为存储内容。

### LUT 结构图（4输入LUT）
- 4输入LUT含16个SRAM单元，输入A0~A3作为地址线，输出存储值。见PPT

### 典型FPGA架构
以Xilinx XC4000为例：
- **CLB**（Configurable Logic Block，可配置逻辑块）：每个CLB包含多个LUT和触发器。
- **IOB**（Input/Output Block，输入输出块）：可编程方向(用户可以通过配置寄存器，把某个引脚单独定义为输入引脚、输出引脚，或者双向（Bidirectional）引脚)、三态(三态门)、速率控制(指控制输出引脚电平跳变（0变1或1变0）的上升沿/下降沿的陡峭程度)。
- **可编程开关矩阵**和**连线资源**（互连通道）。

以Altera Cyclone为例：
- **LE**（Logic Element，逻辑单元）：含一个4输入LUT和一个寄存器，可配置为组合/时序。
- **LAB**（Logic Array Block，逻辑阵列块）：由16个LE组成。
- 全局时钟网络、RAM块（M9K块）、DSP块。

### FPGA 特点
- **易失性**（SRAM工艺），需外挂配置芯片（如EPCS）上电加载。
- **容量大**，可集成数百万逻辑单元。
- **并行处理能力强**，适合复杂算法、高速接口。
- 延时**不可完全预测**（取决于布线），但灵活性极高。

## 2.6 FPGA/CPLD 的编程元件（工艺对比）

| 编程/配置元件 | 工艺类型 | 是否易失 | 擦写/配置次数 | 典型应用 | 核心特点 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **熔丝（Fuse）** | 熔断型 | **非易失** | 1次（OTP） | 早期PROM | 靠大电流烧断金属连线，不可逆 |
| **反熔丝（Antifuse）** | 高压击穿 | **非易失** | 1次（OTP） | 航天/军工FPGA | 靠高压编程使绝缘层击穿形成导通，**抗辐射、可靠性极高** |
| **EPROM** | 紫外线擦除 | **非易失** | 几十~几百次 | 早期PLD | 擦除需紫外光照射（慢），窗口型封装 |
| **EEPROM** | 电擦写 | **非易失** | ~1万次 | **多数CPLD** | 电擦除，**在系统可编程（ISP）**，无需紫外线 |
| **Flash** | 快闪存储 | **非易失** | ~1万次 | 部分CPLD（如MAX II） | 高密度，电擦写，成本低 |
| **SRAM** | 静态存储器 | **易失（掉电丢失）** | **无限次（无磨损）** | **绝大多数FPGA** | 基于CMOS锁存器，**需外挂配置芯片**（如EPCS），上电自动加载 |

## 2.7 边界扫描测试技术（JTAG / IEEE 1149.1）

### 背景
- VLSI（超大规模集成电路）器件引脚多，传统物理探针测试困难。
- **JTAG**（Joint Test Action Group，联合测试行动组）制定**IEEE 1149.1**边界扫描标准。

### 边界扫描原理
- 在每个I/O引脚内部增加一个**边界扫描单元**（含移位寄存器）。
- 所有边界扫描单元串联成一条**扫描链**（从TDI进，TDO出）。
- 可**串行输入/输出**测试数据，控制引脚状态或读取引脚电平。

### JTAG 接口引脚（5线）

| 引脚 | 全称 | 功能 |
|------|------|------|
| **TDI** | Test Data Input | 测试数据串行输入，TCK上升沿采样 |
| **TDO** | Test Data Output | 测试数据串行输出，TCK下降沿输出，高阻态空闲 |
| **TMS** | Test Mode Select | 控制TAP（Test Access Port）状态机跳转 |
| **TCK** | Test Clock Input | 测试时钟，上升沿/下降沿触发 |
| **TRST** | Test Reset Input（可选） | 低电平异步复位边界扫描逻辑 |

### 边界扫描的三大应用
1. **在系统测试**（检测PCB焊接、短路/断路）。
2. **编程/配置下载**（将配置数据通过JTAG写入FPGA/CPLD，即ISP）。
3. **在线调试**（如Altera SignalTap、Xilinx ChipScope，实时采样内部节点）。

### 与编程配置的关系
- JTAG接口也是**ISP**（In-System Programming，在系统可编程）的主要通道。
- 基于IEEE 1149.1，后续制定**IEEE 1532**（配置编程标准），统一了JTAG编程流程。

## 2.8 FPGA/CPLD 的编程与配置

### ISP（In System Programmable，在系统可编程）
- 优点：焊接后仍可编程、现场升级、减少器件损伤、支持样机调试。

### 配置模式（以Xilinx 7系列为例）

## 课后习题答案

### 2.1 PLA和PAL在结构上有什么区别？
- **PLA**：与阵列可编程，或阵列可编程。
- **PAL**：与阵列可编程，或阵列固定。
- 因此PLA更灵活，但PAL速度更快、成本更低。

### 2.2 GAL的OLMC有什么特点？怎样实现可编程组合电路和时序电路？
- **OLMC**（Output Logic Macro Cell）包含MUX、寄存器、异或门等，可配置为：
  - 组合输出：直通或异或输出，无时钟。
  - 时序输出：通过寄存器寄存输出，可编程时钟/使能/复位。
- 通过编程选择OLMC内部MUX的路径，决定反馈来源、输出极性、寄存器使用等，从而实现不同模式。

### 2.3 简述基于乘积项的可编程逻辑器件的结构特点。
- 采用**与-或阵列**结构：与阵列产生乘积项（最小项），或阵列将乘积项相或实现逻辑函数。
- 典型代表：PAL、GAL、CPLD。
- 特点：延时固定、适合组合逻辑和有限状态机，密度中等。

### 2.4 基于查找表的可编程逻辑结构的原理是什么？
- 使用**SRAM存储单元**构成LUT，预先存储所有输入组合对应的输出值。
- 输入信号作为地址，输出对应存储单元的值，即可实现任意逻辑函数。
- LUT规模取决于输入数（如4-LUT有16个存储单元）。

### 2.5 基于乘积项和基于查找表的结构各有什么优点？
- **乘积项**：延时可预测，非易失，适合控制逻辑、状态机，抗干扰强。
- **查找表（LUT）**：灵活性强，可实现任意复杂组合逻辑，容量大，适合数据密集型、算法型应用。

### 2.6 CPLD和FPGA在结构上有什么明显区别？各有何特点？
| 对比 | CPLD | FPGA |
|------|------|------|
| 结构 | 乘积项（与-或阵列）+ 宏单元 | 查找表（LUT）+ 触发器 + 互连 |
| 工艺 | EEPROM/Flash（非易失） | SRAM（易失，需配置芯片） |
| 密度 | 低至中等（数百~数千宏单元） | 高（数万~数百万逻辑单元） |
| 延时 | 可预测、恒定 | 与布局布线相关，不可完全预测 |
| 适用 | 控制逻辑、胶合逻辑 | 复杂算法、高速接口、数字信号处理 |
| 功耗 | 静态功耗较低 | 动态功耗较高，静态漏电偏大 |

### 2.7 FPGA器件中的存储器块有何作用？
- 提供**片上RAM/ROM**（如块RAM，BRAM），用于存储数据、FIFO、缓存、查找表、系数等。
- 可配置为单口/双口RAM、ROM、移位寄存器等，提高系统集成度和性能。

### 2.8 Altera的MAX II器件是属于CPLD还是FPGA？
- **MAX II** 属于**CPLD**（尽管采用LUT结构，但内部有非易失Flash配置，上电即用，延时固定，归类为CPLD）。实际是“CPLD-like”架构。

### 2.9 边界扫描技术有什么优点？
- 无需物理探针，通过JTAG串行接口可测试PCB互连、芯片内部逻辑。
- 支持**在系统编程**（ISP）和**在线调试**（如SignalTap）。
- 减少测试设备成本，提高测试覆盖率和生产效率。

### 2.10 JTAG接口都有哪些功能？
- 边界扫描测试（检测PCB焊接/开路/短路）。
- 编程/配置FPGA/CPLD（下载配置文件）。
- 在线调试（实时捕获内部信号，如逻辑分析仪）。
- 符合IEEE 1149.1标准，可级联多器件。

### 2.11 FPGA/CPLD器件未来的发展趋势有哪些？
- 更高集成度（更多逻辑、DSP、RAM、收发器）。
- 更低电压和功耗（如1.0V以下内核）。
- 嵌入硬核处理器（ARM、RISC-V）及AI加速器。
- 支持更高速串行接口（56G/112G PAM4）。
- 向异构计算、3D堆叠、模数混合方向发展。

### 2.12 FPGA/CPLD器件之间的区别

| 区别 | **FPGA** | **CPLD** |
|------|----------|----------|
| 原理 | 查找表（LUT）+ 触发器 | 乘积项（与-或阵列）+ 宏单元 |
| 工艺 | SRAM（部分Flash/反熔丝） | EEPROM / Flash |
| 掉电是否数据丢失 | **是**（SRAM型），需外挂配置芯片 | **否**（非易失） |
| 是否需要外挂配置芯片 | **是**（SRAM型必须） | **否**（片上存储配置） |
| 集成度（高或低） | **高**（数十万~数百万门） | **低**（数千~数万门） |
| 价格（高或低） | **高**（单位逻辑价格低，但总价高） | **低**（小规模低成本） |

# 第4章 Verilog HDL设计初步（简略）

## 4.1 设计举例（直观感受）
- **4位全加器**：使用 `assign` 连续赋值语句，组合逻辑。
  ```verilog
  module adder4(cout,sum,ina,inb,cin);
    output[3:0] sum;
    output cout;
    input[3:0] ina,inb;
    input cin;
    assign {cout,sum}=ina+inb+cin;
  endmodule
  ```
- **4位计数器**：使用 `always` 过程块，时序逻辑（同步复位）。
  ```verilog
  module count4(out,reset,clk);
    output[3:0] out;
    input reset,clk;
    reg[3:0] out;
    always @(posedge clk) begin
      if(reset) out <= 0;
      else out <= out + 1;
    end
  endmodule
  ```


## 4.2 模块结构（模板）
- **端口声明**：`input`、`output`、`inout`。
- **数据类型声明**：`wire`、`reg` 等。
- **逻辑描述**：
  - `assign` 连续赋值（组合逻辑）。
  - `always` 过程块（时序/组合逻辑）。
  - 模块例化（调用其他模块）。
  - 门元件例化。

## Verilog 程序的5大特点
1. **模块化**：由 `module` ... `endmodule` 构成。每个模块实现特定的功能；模块是可以进行层次嵌套。
2. **端口定义**：必须先声明输入/输出方向。
3. **书写自由**：一行可多语句，一语句可多行。
4. **分号结尾**：除 `endmodule` 等少数语句外，每条语句以 `;` 结束。
5. **注释**：`//` 单行，`/* */` 多行。

# 第5章 Verilog HDL语法与要素

## 5.1 语言要素（符号集）

| 要素 | 说明 | 注意点 |
| :--- | :--- | :--- |
| **空白符** | 空格、Tab、换行、换页 | 综合时被忽略，仅用于提高可读性 |
| **注释** | `//` 单行，`/* */` 多行 | 单行不允许续行；多行可跨行 |
| **标识符** | 字母、数字、`$`、`_` 组合 | **首字符必须是字母或下划线**；区分大小写（`count` ≠ `COUNT`） |
| **关键字** | 如 `module`、`always`、`initial` | **全部小写**，用户不能用作变量名 |

---

## 5.2 常量（Constants）
### 1. 整数（Integer）
- **格式**：`+/- <位宽> '<进制> <数字>`
- **进制**：`b/B`（二进制）、`d/D`（十进制）、`h/H`（十六进制）、`o/O`（八进制）。缺省进制为十进制。
- **合法示例**：
  - `8'b11000101`
  - `8'hd5`
  - `4'B1x_01`（下划线可提高可读性，被忽略）
  - `5'Hx` → 扩展为 `xxxxx`
  - `4'hZ` → 扩展为 `zzzz`
- **非法示例**：`8' h2A`（位宽和进制间**不允许**有空格，但进制和数字间可以）；`8'h2 A`（数字间不允许空格）。

- 因此可以理解为：**最前面的位宽数字**是一个整体，**('h)** 是一个整体，最后的数字**d5** 是一个整体。整体与整体之间可以有空格，但是整体内部不允许有空格。
- **注意**：`x`（不定态）、`z`（高阻态）可用于整数常量表示。
### 2. 实数（Real）
- **十进制表示**：必须**小数点两侧都有数字**，如 `2.0`、`0.1`，`2.` 非法。
- **科学计数法**：`43_5.1e2` → 43510.0；`5E-4` → 0.0005。

### 3. 字符串（String）
- 双引号内的字符序列，**不能分多行**。主要用于仿真显示信息。

## 5.3 数据类型（Data Type）

### 4种基本逻辑状态（Verilog 所有类型均基于此）
| 值 | 含义 |
| :--- | :--- |
| **0** | 低电平、逻辑假 |
| **1** | 高电平、逻辑真 |
| **x (X)** | 不确定/未知逻辑状态 |
| **z (Z)** | 高阻态（三态） |

> `x` 和 `z` 不区分大小写。

### 数据分类（两大派系）

| 类型分类 | 对应硬件 | 典型关键字 | 赋值方式 | 存储特性 |
| :--- | :--- | :--- | :--- | :--- |
| **net型（连线型）** | 物理连线（wire） | `wire`、`tri` | 用 `assign` 连续赋值，或连接门/模块输出 | **输出随输入变化**，无存储 |
| **variable型（变量型）** | 不一定对应寄存器，视综合情况而定 | `reg`、`integer` | 在 `always` / `initial` 过程块中赋值 | 有存储行为（过程赋值保持值） |

> **重要**：`reg` 不一定是硬件寄存器，综合器会根据使用方式映射为连线、触发器或锁存器。

**定义格式**：
- `wire [7:0] databus;` // 8位wire向量

## 5.4 参数（Parameter）

- 用 `parameter` 定义**符号常量**，常用于定义时延、位宽等。有点类似于C语言中的 `#define PI 3.1415926`
- **格式**：`parameter 参数名1 = 表达式1, 参数名2 = 表达式2;`
- **示例**：`parameter sel = 8, code = 8'ha3;`

## 5.5 向量（Vector）

### 标量 vs 向量
- **标量**：位宽为1的变量（默认）。
- **向量**：位宽 > 1。定义格式：`[msb : lsb]`（最高位 : 最低位）。
  - 如 `reg [7:0] data;` // 8位向量，data[7] 为最高位。

### 位选择与域选择
- **位选择**：选取向量中的1位，如 `data[3]`。
- **域选择**：选取连续的多位，如 `data[7:4]`。
- **示例**：
  ```verilog
  reg [7:0] a, b;
  reg [3:0] c;
  reg d;
  d = a[7] & b[7];      // 位选择
  c = a[7:4] + b[3:0];  // 域选择
  ```

## 6.1 过程语句（`initial` 与 `always`）

### `initial` 语句
- **执行次数**：**只执行一次**。
- **用途**：仿真时的初始化、生成激励波形（如时钟、复位信号）。
- **可综合性**：**不可综合**，仅用于仿真测试平台（Testbench）。

### `always` 语句
- **执行次数**：**不断重复执行**。
- **用途**：描述组合逻辑和时序逻辑。
- **触发方式**：由**敏感信号列表**（事件表达式）控制。

**基本模板**：
```verilog
always @(<敏感信号表达式>) begin
    // 过程赋值
    // if-else, case 等
end
```

### 敏感信号列表（Event Expression）
- **电平触发**（组合逻辑）：`@(a or b or c)`，当列表中任一信号变化时执行。
- **边沿触发**（时序逻辑）：
  - `@(posedge clk)`：时钟上升沿触发。
  - `@(negedge clk)`：时钟下降沿触发。
  - `@(posedge clk or negedge reset)`：时钟上升沿或复位下降沿触发。
- **组合逻辑完整写法**：必须列出所有输入信号，否则综合时会生成不期望的锁存器（Latch）。

**示例（4选1数据选择器）**：
```verilog
always @(in0 or in1 or in2 or in3 or sel) begin
    case(sel)
        2'b00: out = in0;
        2'b01: out = in1;
        2'b10: out = in2;
        2'b11: out = in3;
        default: out = 1'bx;
    endcase
end
```

**同步与异步复位的区别（重要考点）**：
- **同步复位**：复位信号只在时钟有效边沿到来时才起作用。敏感列表**只含时钟**。
  ```verilog
  always @(posedge clk) begin
      if(!reset) out <= 0;  // 同步复位
      else if(load) out <= data;
      else out <= out + 1;
  end
  ```
- **异步复位**：复位信号**立即**生效，不依赖时钟。敏感列表**同时包含时钟和复位**。
  ```verilog
  always @(posedge clk or negedge reset) begin  // 低有效异步复位
      if(!reset) out <= 0;
      else out <= out + 1;
  end
  ```

## 6.2 块语句（`begin-end` 与 `fork-join`）

| 块类型 | 关键字 | 执行方式 | 可综合性 |
| :--- | :--- | :--- | :---: |
| **串行块** | `begin ... end` | 语句**顺序执行**（从上到下） | ✅ |
| **并行块** | `fork ... join` | 语句**同时执行** | ❌（仅仿真） |

**串行块示例**：
```verilog
begin
    regb = rega;
    regc = regb;
end
// 执行后 regb 和 regc 均等于 rega（regb先更新，regc再取新值）
```

## 6.3 赋值语句

### 1. 持续赋值（Continuous Assignment）—— `assign`
- **适用对象**：`wire` 型变量（net型）。
- **执行特性**：**实时更新**。右端表达式中任一信号变化，立即重新计算并赋给左端。
- **位置**：**不能**放在 `always` 块内部，必须写在模块的顶级作用域。
- **示例**：`assign c = a & b;`

### 2. 过程赋值（Procedural Assignment）—— 用于 `always` / `initial` 内部
赋值对象必须是 `reg` 型变量（variable型）。

#### （1）阻塞赋值（Blocking Assignment）—— `=`
- **执行方式**：**立即执行**。当前语句执行完毕，左端变量值立刻更新，然后才执行下一条语句。
- **特点**：后续语句的执行会被“阻塞”，直到当前赋值完成。
- **适用场景**：**组合逻辑**建模。

#### （2）非阻塞赋值（Non-blocking Assignment）—— `<=`
- **执行方式**：**并行执行**。在过程块（`always`）结束时，所有非阻塞赋值同时更新。
- **特点**：在块执行期间，右端表达式先全部计算并暂存，块结束时统一赋给左端。
- **适用场景**：**时序逻辑**建模（如寄存器、计数器）。

### 阻塞与非阻塞的核心区别（考试必考）

| 对比维度 | 阻塞赋值 `=` | 非阻塞赋值 `<=` |
| :--- | :--- | :--- |
| 更新时机 | **语句结束立即更新** | **过程块结束时统一更新** |
| 后续语句影响 | 后续语句**能看到**更新后的值 | 后续语句**看不到**更新后的值（仍用旧值） |
| 对应硬件 | 组合逻辑（或纯导线） | 时序逻辑（触发器/寄存器） |
| 综合后结构 | 无寄存器（或组合反馈） | 生成触发器 |

**典型反例对比（波形不同）**：
- **非阻塞**（`b<=a; c<=b;`）：在时钟上升沿，同时将 a 赋给 b，b 的旧值赋给 c，即 **c 落后 b 一个周期**。
- **阻塞**（`b=a; c=b;`）：在时钟上升沿，先更新 b 为 a，再更新 c 为新的 b，**c 与 b 在同一周期同时等于 a**。

### ★ 阻塞/非阻塞赋值使用原则

1. **组合逻辑**（`always @(*)`）用**阻塞赋值 `=`**。
2. **时序逻辑**（`always @(posedge clk)`）用**非阻塞赋值 `<=`**。
3. **锁存器建模**用非阻塞赋值。
4. 同一 `always` 块中同时含组合和时序逻辑时，推荐**非阻塞赋值**。
5. **禁止在同一 `always` 块中混合使用** `=` 和 `<=`（综合器可能报错或产生竞争）。
6. **严禁在两个或多个 `always` 块中对同一变量赋值**（多驱动冲突，综合报错）。
- **注意**：组合逻辑：敏感列表推荐使用@(*)避免漏信号产生锁存器）。时序逻辑：敏感列表必须以时钟边沿（posedge clk）为核心，严禁混入 * 作为普通电平触发，即使语法允许，综合器也会忽略且导致前仿后仿不一致。

## 6.4 条件语句

### `if-else` 语句
- 语法与 C 语言完全一致，支持三种形式：
  - `if (条件) 语句;`
  - `if (条件) 语句1; else 语句2;`
  - `if (条件1) 语句1; else if (条件2) 语句2; ... else 语句n;`
- **注意**：`if-else` 可能产生优先编码器结构（嵌套多级），而 `case` 通常生成多路选择器。

### `case` 语句
- **格式**：
  ```verilog
  case (敏感表达式)
      值1: 语句1;
      值2: 语句2;
      ...
      default: 语句n;
  endcase
  ```
- **分支**：各分支值必须互不重叠。
- **`default`**：用于避免锁存器生成（建议总是加上）。

### 特殊 case 变体（仿真用途，综合慎用）

| 语句 | 特性 | 综合支持 |
| :--- | :--- | :---: |
| `case` | 精确匹配（需完全相等） | ✅ |
| `casez` | 将 `z` 视为**无关项**（不比较 `z` 位） | 部分支持 |
| `casex` | 将 `x` 和 `z` 都视为无关项 | 不推荐综合 |

**七段数码管译码器示例**：用 `case` 将 0~9 映射到 7 位段码。

## 6.5 循环语句

| 循环类型 | 关键字 | 可综合性 | 主要用途 |
| :--- | :--- | :---: | :--- |
| `for` | `for` | ✅（条件可综合） | 描述重复结构（如移位、加法树、投票计数器） |
| `repeat` | `repeat` | ❌（通常仿真） | 固定次数循环 |
| `while` | `while` | ❌（通常仿真） | 条件控制循环 |
| `forever` | `forever` | ❌（仅仿真） | 生成周期性激励（如时钟） |

**可综合的 `for` 示例（七人投票表决器）**：
```verilog
always @(vote) begin
    sum = 0;
    for(i = 0; i <= 6; i = i + 1)  // 循环展开为7个加法器
        if(vote[i]) sum = sum + 1;
    pass = (sum[2]) ? 1 : 0;  // 超过4票（sum>3）则通过
end
```
> 综合时，`for` 循环会被**展开**为并行硬件，循环次数必须是常数或可静态确定。

## 6.7 任务（Task）与函数（Function）

### 任务（Task）
- **定义格式**（无端口列表）：
  ```verilog
  task <任务名>;
      端口及数据类型声明;  // input / output / inout
      局部变量声明;
      语句;
  endtask
  ```
- **调用**：`<任务名>(端口1, 端口2, ...);`
- **特性**：
  - 可有任意个输入、输出、双向端口。
  - **支持** `#`、`@`、`wait` 等时序控制。
  - 可调用其它任务和函数。
  - **无返回值**（通过 `output` 端口传递结果）。
  - 只能在 `always` / `initial` 过程块中调用（**不能**出现在 `assign` 中）。

### 函数（Function）
- **定义格式**：
  ```verilog
  function <位宽或类型> 函数名;
      端口声明;  // 仅允许 input
      局部变量;
      语句;
      函数名 = 返回值;  // 通过函数名本身赋值返回
  endfunction
  ```
- **调用**：作为表达式的一部分，例如 `assign y = get0(x);`
- **特性**：
  - **至少有一个输入**，不允许 `inout` 或 `output`。
  - **不支持**时序控制（`#`、`@`、`wait`）。
  - 可调用其它函数，但**不能调用任务**。
  - 可在 `assign` 持续赋值语句中调用（在右端表达式）。

### 任务与函数对比表

| 比较项目 | **任务（Task）** | **函数（Function）** |
| :--- | :--- | :--- |
| **输入/输出** | 可有任意个 `input`、`output`、`inout` | **至少有一个 `input`**，无 `output` / `inout` |
| **返回值** | **无**返回值，通过 `output` 传递结果 | **有**返回值，通过函数名返回 |
| **调用位置** | 仅在过程语句（`always`/`initial`）中调用 | 可在过程语句和 `assign` 连续赋值中调用 |
| **时序控制** | 可包含 `#`、`@`、`wait` | **不可包含**任何时序控制语句 |
| **调用其它** | 可调用其它任务和函数 | 可调用其它函数，**不可调用任务** |

## 6.8 顺序执行与并发执行（硬件执行机制）

Verilog 的执行模型严格区分**并发（Parallel）**与**顺序（Sequential）**，这取决于语句类型和所在的代码层级。

### 1. 并发执行（Parallel）
- **模块级并行**：多个 `always` 块、`assign` 持续赋值语句、模块实例化之间，在仿真起始时**同时开始执行**，相互独立，顺序无关。
- **过程块内部并行（非阻塞赋值）**：在同一个 `always` 块中，非阻塞赋值（`<=`）在**同一个仿真时间步（即同一个时钟边沿）内并行完成**。
  - **执行分两步**：
    1. **采样（右式计算）**：在块开始执行的时刻，立即读取所有右端表达式（RHS）的**当前值**并暂存。
    2. **更新（左式赋值）**：当整个 `always` 块中的所有语句都执行完**采样**后，再将暂存的值**同时**赋给左侧变量（LHS）。
  - 因此，非阻塞赋值不存在“先后依赖”，所有赋值动作发生在**同一时刻**。

### 2. 顺序执行（Sequential）
- **模块内部顺序**：`begin-end` 串行块中的语句**自上而下依次执行**。
- **过程块内部顺序（阻塞赋值）**：在 `always` / `initial` 块中，阻塞赋值（`=`）**立即完成计算并更新**左端变量。只有当前语句执行完毕，才会开始执行下一条语句，后续语句能立即看到前一条语句的更新结果。


### 3. 经典对比示例（底层执行步骤详解）

设初始值为：`q = 0`，`a = 1`。

#### 场景一：阻塞赋值（`=`），顺序影响最终结果

- **模块1（q 先更新，a 后更新）**：
  ```verilog
  begin
      q = ~q;   // 步骤1：q 立刻变为 ~0 = 1
      a = ~q;   // 步骤2：此时 q 已为 1，所以 a 变为 ~1 = 0
  end
  // 最终结果：q = 1, a = 0（综合为串联反相器链）
  ```

- **模块2（a 先更新，q 后更新）**：
  ```verilog
  begin
      a = ~q;   // 步骤1：此时 q 仍为 0，所以 a 变为 ~0 = 1
      q = ~q;   // 步骤2：此时 q 仍为 0，所以 q 变为 ~0 = 1
  end
  // 最终结果：q = 1, a = 1（综合为单个反相器分叉驱动）
  ```

> **结论**：阻塞赋值时，**书写顺序直接决定综合后的硬件拓扑结构和波形**。


#### 场景二：非阻塞赋值（`<=`），采样与更新分离，顺序无关

初始值仍为：`q = 0`，`a = 1`。

- **写法 1（q 在前，a 在后）**：
  ```verilog
  begin
      q <= ~q;   // ① 仅采样：暂存 ~0 = 1（q 暂不更新）
      a <= ~q;   // ② 仅采样：暂存 ~0 = 1（此时 q 仍未更新，取旧值 0）
                 // ③ 块结束：同时将暂存值 1 赋给 q 和 a
  end
  // 最终结果：q = 1, a = 1
  ```

- **写法 2（a 在前，q 在后）**：
  ```verilog
  begin
      a <= ~q;   // ① 仅采样：暂存 ~0 = 1
      q <= ~q;   // ② 仅采样：暂存 ~0 = 1（此时 q 仍未更新）
                 // ③ 块结束：同时将暂存值 1 赋给 q 和 a
  end
  // 最终结果：q = 1, a = 1（与写法1完全相同）
  ```

> **结论**：非阻塞赋值时，**书写顺序不影响最终结果**。因为所有右端表达式的值都在块执行前被“拍快照”锁存，更新动作统一在块结束时发生。

### 4. 核心总结

- **并发（同时）** 指的是 **“同一个仿真时间步结束时的统一更新”**，而非微观上的绝对零延时。
- **顺序** 指的是 **“语句间的即时可见性”**，阻塞赋值会立刻将结果暴露给后续语句。
- **判定原则**：
  - 看到 `<=` 且位于 `always @(posedge clk)` 中 → **并行（同时更新，顺序无关）**。
  - 看到 `=` 且位于 `begin-end` 串行块中 → **顺序（自上而下，依赖顺序）**。

# 第7章 数字设计的层次与风格 - 笔记整理（无代码版）

## 1. 三种描述风格总览（本章核心框架）

Verilog 对同一硬件功能有三种表达方式，抽象层次由低到高排列：

| 描述风格 | 关键字/核心机制 | 抽象层次 | 设计效率 | 对综合器的优化空间 |
| :--- | :--- | :--- | :--- | :--- |
| **结构化描述（门级）** | 调用内置门元件（`and`、`or`、`not` 等） | **最低**（接近物理实现） | 低（繁琐） | 小（网表已固定） |
| **数据流描述（RTL级）** | `assign` 持续赋值语句 | **中等**（逻辑表达式） | 较高 | 中 |
| **行为化描述（算法级）** | `always` 过程块（含 `if-else`、`case`、循环等） | **最高**（算法/功能） | 高（类似C语言） | **最大**（综合器可充分优化） |

> **设计准则**：除非是对关键路径（时序要求极高）进行手工门级调优，工程上**优先采用行为描述**，把底层电路优化交给综合器。

---
## 2. 结构化描述（Structural / Gate-Level）

**本质**：像搭建积木一样，通过实例化（调用）Verilog 预定义的内置门元件来连接成电路。

- **硬件映射**：直接对应具体的门电路（与门、或门、非门、三态门等）。
- **书写特征**：声明内部连接线（`wire`），然后列出元件实例。
- **分类**：
  - **多输入门**：`and`、`nand`、`or`、`nor`、`xor`、`xnor`（端口顺序：输出, 输入1, 输入2, ...）。
  - **多输出门**：`buf`、`not`（端口顺序：输出, 输入）。
  - **三态门**：`buff1`（高电平使能）、`buff0`（低电平使能）、`notif1`、`notif0`（端口顺序：输出, 输入, 使能控制端）。
- **特点**：可读性最差，设计大型电路极容易出错，但在底层接口或时序精确调整时仍会使用。

---
## 3. 数据流描述（Data Flow / RTL-Level）

**本质**：通过**持续赋值语句 `assign`** 直接描述输入与输出之间的逻辑表达式或条件运算。

- **硬件映射**：对应组合逻辑网络（纯硬件连线），右端操作数任一变化，左端立即重新计算更新。
- **赋值对象**：**必须是 `wire`（net型）变量**。
- **特点**：简洁高效，介于门级和行为级之间，是RTL（寄存器传输级）视图的直接体现。
- **书写特征**：大量使用 `assign` 和三目运算符（`? :`），如 `assign out = sel ? b : a;`。
- **考试要求**：**不要求会写复杂模块**，但要能看懂 `assign` 语句表达的逻辑功能，并能与行为级写法相互转换（思路）。

---
## 4. 行为化描述（Behavioral / Algorithmic）⭐ 

**本质**：使用 **`always` 过程块**，通过高级语言风格的语句（`if-else`、`case`、`for` 等）来描述电路**在给定输入下应作出何种反应**。

- **硬件映射**：综合器将其翻译为具体电路（组合逻辑、时序逻辑或二者的混合）。
- **赋值对象**：过程块内被赋值的变量**必须是 `reg` 型（variable型）**。
- **核心优势**：
  - 抽象程度最高，设计者只需关心功能，无需操心门级连接。
  - 给综合器留下了最大的优化空间（综合器可以重排逻辑、资源共享等）。
- **书写特征**：包含 `always @(...)` 敏感列表。

> **一句话分辨**：有 `assign` 就是数据流，有 `always` 就是行为级，有一堆 `and/or` 实例化就是结构级。

---
## 5. 三种描述风格的综合对比（以同一个2选1MUX为例）

- **结构级**：手动例化 `not`、`and`、`or` 三个门，搭出与或表达式。
- **数据流级**：一行 `assign out = (sel) ? b : a;`
- **行为级**：在 `always` 块中用 `if (sel) out = b; else out = a;`

**考试易考点**：给出某模块代码，让你判断它是哪种描述风格。

---
## 6. 基本组合电路设计（内容梳理）

本章列举了几类典型组合逻辑模块，要求能通过阅读明确其功能：

1. **多路选择器（MUX）**：
   - 可以通过**门级**、`assign` **三目运算符**、`always` 块中的 **`if-else`** 或 **`case`** 实现。
   - `case` 语句描述译码器或选择器时结构最清晰。

2. **译码器（Decoder）**：
   - 典型的 3-8 译码器，使用 `case` 语句将输入的 3 位二进制数映射到 8 位独热码输出（低电平有效或高电平有效）。

3. **奇偶校验位发生器**：
   - 利用**缩位异或运算符（`^`）**对向量所有位进行归约，得到偶校验位；再取反得到奇校验位。
   - 校验规则：偶校验保证传输数据中“1”的个数为偶数。

4. **三态门（Tri-state Buffer）**：
   - 通过 `buff1`（高电平导通）或 `buff0`（低电平导通）实现总线隔离。

## 7. 基本时序电路设计
### 7.1 D触发器（DFF）的不同写法对比

| 触发器类型 | 敏感列表特征 | 复位/置位生效时机 | 工程常用场景 |
| :--- | :--- | :--- | :--- |
| **基本D触发器** | 仅 `posedge clk` | 无复位 | 最基础寄存器 |
| **同步复位/置位** | 仅 `posedge clk` | 只有在时钟上升沿到来时，检测到复位/置位信号才生效 | 利于时序分析，推荐使用 |
| **异步复位/置位** | `posedge clk or negedge reset`（含复位/置位沿） | 复位/置位信号**立即生效**，不依赖时钟 | 需要快速响应的紧急控制逻辑 |

> **填空/简答**：异步复位敏感列表必须包含 **复位信号的边沿**（`posedge` 或 `negedge`），且赋值语句写在 `if` 条件中，位于所有其他逻辑之前。

### 7.2 锁存器（Latch） vs 寄存器（Register）

| 对比维度 | **锁存器（Latch）** | **寄存器（Register/FF）** |
| :--- | :--- | :--- |
| **敏感方式** | **电平敏感**（如 `always @(clk or data)`，当 `clk` 为高电平时透明） | **边沿敏感**（如 `always @(posedge clk)`） |
| **触发时刻** | 控制信号（如时钟）处于有效电平的**整个期间**都跟随输入变化 | 仅在时钟**上升沿或下降沿那一瞬间**采样输入 |
| **硬件结构** | 通常由门电路构成（面积小） | 由主从触发器构成（面积大，但时序可控） |
| **设计风险** | **容易产生毛刺和竞争冒险**，在FPGA中应尽量避免（综合器在组合逻辑敏感列表不完整时自动推断出Latch，通常是设计错误） | 时序稳定，是同步设计的基石 |
| **使用原则** | 尽量避免；只有在极其特殊的数据锁存场合（如地址锁存器）才使用。**如果你在 `always` 中没有列出所有分支（缺少 `else`），综合器会推断出Latch，这通常不是你想要的！** | 广泛使用，推荐 |

### 7.3 计数器（Counter）
- 典型结构：带**同步清零**、**同步预置**（加载初值）、**加减控制**。
- 注意敏感列表通常**只含时钟边沿**（同步设计），所有判断都在块内完成。

# 第8章 Verilog有限状态机设计


## 1. 有限状态机（FSM）概述

**定义**：由**一组状态**、**一个输入集合**和**一个输出集合**组成的时序电路。电路的输出取决于当前状态（以及输入，取决于类型）。

### 1.1 状态机的两大类型

| 类型 | 输出取决于 | 特点 |
| :--- | :--- | :--- |
| **Moore型** | **仅取决于当前状态** | 输出与输入无关，时序稳定，不易产生毛刺。输出在状态转换完成后立即稳定 |
| **Mealy型** | **取决于当前状态 + 当前输入** | 输出对输入敏感，响应更快，但可能存在毛刺。输出在输入变化时可能立即变化（组合逻辑直接产生） |

> **考试判断方法**：看输出赋值语句的敏感列表是否包含输入信号。若只含 `state`（现态）→ Moore型；若包含 `state` 和输入（如 `x`）→ Mealy型。

### 1.2 状态机的基本组成（三要素）

| 组成模块 | 功能 | 逻辑类型 |
| :--- | :--- | :--- |
| **状态寄存器** | 存储当前状态（`state`），每个时钟边沿更新为下一状态（`next_state`） | 时序逻辑（触发器） |
| **次态逻辑（NS Logic）** | 根据当前状态和输入，计算下一状态 | 组合逻辑 |
| **输出逻辑（OL Logic）** | 根据当前状态（Moore）或当前状态+输入（Mealy）计算输出 | 组合逻辑 |

---
## 2. 状态转换图的画法 ⭐

状态转换图（State Transition Diagram，STD）是设计状态机**第一步**，必须掌握规范画法。

### 2.1 图形规范

| 图形元素 | 表示方式 | 说明 |
| :--- | :--- | :--- |
| **状态** | **圆角矩形**或**圆圈**，内部填写状态名（如 `S0`、`IDLE`） | 每个状态有唯一的标识符 |
| **初始状态** | 带**双圈**或**箭头指向** | 表示复位后进入的起始状态 |
| **状态转移** | **有向箭头**（→）从一个状态指向另一个状态 | 箭头上标注**转移条件** |


### 2.2 转移条件的标注格式

- **格式**：`条件表达式 / 输出值`（Mealy型）或只写条件（Moore型输出写在状态框内）。
- **规则**：
  - 条件通常为输入信号的组合（如 `x=1`、`(x==1)`、`en`）
  - 多个条件分支：标出 `条件1`、`条件2`（如 `x=0` / `x=1`）
  - 无条件转移：标注 `1`（恒真）
  - 条件互斥（同一输入不能同时满足两个条件）

### 2.3 四步画图法（以101序列检测器为例）

1. **确定状态含义**：定义每个状态代表“已经检测到的序列前缀长度”。
   - S0：初始/未检测到任何有效位
   - S1：已收到 `1`（前缀“1”）
   - S2：已收到 `10`（前缀“10”）
   - S3：已收到 `101`（完整序列，输出为1，可重叠检测）

2. **画出所有状态节点**：用圆框排列。

3. **画出初始状态指示**：S0 为起始状态，标注复位箭头。

4. **逐一添加转移箭头**：
   - 从当前状态出发，列出所有可能的输入取值，确定下一状态。
   - 标注转移条件。
   - 检查是否所有状态的所有输入组合都有明确的转移方向（**完备性**）。

### 2.4 状态转换图的完备性检查

- **每个状态必须对所有可能的输入组合都有明确的转移目标**。
- 若有未定义的输入组合，必须用 `default` 或显式分支将其导向已知状态（通常导向起始状态），避免综合出锁存器。

## 3. 状态机的四种描述方式（按always块数量划分）

同一状态机可以用不同数量的 `always` 过程块来描述。块数越多，逻辑层次越清晰，但代码量越大。注意区分四种写法。

### 3.1 三过程描述（最规范，代码可读性最高）

| always块 | 职责 | 逻辑类型 | 赋值方式 |
| :--- | :--- | :--- | :--- |
| **过程1（时序）** | 状态寄存器更新：`state <= next_state` | 时序逻辑 | 非阻塞 `<=` |
| **过程2（组合）** | 次态逻辑：根据 `state` 和输入计算 `next_state` | 组合逻辑 | 阻塞 `=` |
| **过程3（组合）** | 输出逻辑：根据 `state`（或 `state`+输入）计算输出 `z` | 组合逻辑 | 阻塞 `=` |

- **优点**：三个功能模块完全分离，易于修改和调试。
- **敏感列表写法**：
  - 时序块：`always @(posedge clk or posedge clr)`
  - 次态组合块：`always @(state or x)` 或 `always @(*)`
  - 输出组合块：`always @(state)`（Moore）或 `always @(state or x)`（Mealy）

### 3.2 双过程描述

有两种常见变体：

| 变体 | 块1（时序） | 块2（组合） |
| :--- | :--- | :--- |
| **变体A**（CS+NS / OL） | 状态寄存器更新（`state <= next_state`） | **次态+输出逻辑**合并计算 |
| **变体B**（CS / NS+OL） | 状态寄存器更新 | **次态+输出逻辑**合并计算（另一种划分） |



### 3.3 单过程描述（代码最紧凑，但不推荐大工程）

- **结构**：所有逻辑（状态寄存、次态计算、输出计算）写入**同一个 `always` 时序块**（含时钟边沿敏感）。
- **特点**：
  - 输出和状态更新在同一时钟边沿完成。
  - 由于 `always` 块内为时序逻辑，状态和输出通常使用**非阻塞赋值 `<=`**。
  - 代码紧凑，但状态和输出耦合在一起，可读性稍差。
- **注意**：输出（`z`）在单过程描述中通常用非阻塞赋值（`<=`）或者在块内使用阻塞赋值（`=`）但紧跟在状态更新前，需要理解其与多过程描述的时序差异（输出会延迟一拍）。

### 3.4 四种描述方式的选择原则
| 描述方式 | 代码风格 | 推荐场合 |
| :--- | :--- | :--- |
| **三过程** | 结构清晰，3个always | **工程项目首选**，便于维护和修改 |
| **双过程** | 较紧凑，2个always | 小型状态机，希望减少代码量 |
| **单过程** | 最紧凑，1个always | 非常简单的状态机（如习题、考试），但可读性最差 |

## 4. 状态编码方式

状态编码决定了状态寄存器中每个状态对应的二进制码字，影响电路的速度、面积和功耗。

### 4.1 四种常用编码方式

| 编码方式 | 原理 | 状态数N的码长 | 特点 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **顺序编码**（Binary） | 状态按二进制自然顺序编码（S0=00, S1=01, S2=10, S3=11） | `ceil(log2(N))` | 码长最短，寄存器最少；但相邻状态转移时可能有多位同时翻转，产生**竞争冒险** | 状态数较多、对面积敏感的设计 |
| **格雷编码**（Gray） | 相邻状态码字**仅1位不同** | `ceil(log2(N))` | 状态转移时只有1位翻转，**功耗低、毛刺小**；但码字利用率低 | 高速时序、低功耗设计 |
| **约翰逊编码**（Johnson） | 移位寄存器型编码，每个状态码字由前一个状态左移产生 | N | 译码逻辑简单（只需检测连续0或1）；但状态数受限 | 计数器、地址译码器 |
| **一位热码**（One-Hot） | 每个状态独占1位，有效位为1 | N | **译码逻辑最简单**（判断状态只需检查1位），时序性能最好；但寄存器数量最多（N个触发器） | **FPGA设计中最常用**（FPGA触发器资源丰富，逻辑资源相对紧缺，用触发器换逻辑是划算的） |

### 4.2 四种编码方式对比

| 编码方式 | 码长 | 状态转移时的位翻转数 | 译码逻辑复杂度 | 功耗 | 适用器件 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 顺序编码 | 最短 | 最多（平均N/2位） | 较复杂 | 较高 | ASIC（面积敏感） |
| 格雷编码 | 最短 | 恒为1位 | 较复杂 | 最低 | 高速/低功耗 |
| 约翰逊编码 | N | 1位 | 简单 | 低 | 计数器类 |
| 一位热码 | N | 2位（旧位清0，新位置1） | **最简单**（每位对应一个状态） | 中等（但无毛刺） | **FPGA**（推荐） |

> **问答题**：为何FPGA设计中一位热码最常用？答：FPGA拥有大量触发器资源，一位热码用触发器换译码逻辑，可提高时钟频率、减少组合逻辑延时，且状态转移时无毛刺。

## 5. 状态编码的定义方法（`parameter` vs `localparam` vs `` `define ``）

在Verilog中定义状态码字有三种方式。

### 5.1 三种定义方式对比

| 对比项 | `parameter` | `localparam` | `` `define `` |
| :--- | :--- | :--- | :--- |
| **作用范围** | 本模块内（可跨模块通过实例化传递） | 本模块内（不可传递） | **整个工程**（跨模块，直到遇 `` `undef `` 失效） |
| **是否可被上层重定义（参数传递）** | **可以**（通过 `#(参数)` 或 `defparam` 重定义） | **不可以**（局部参数，不可覆盖） | 不可以（宏替换，全局文本） |
| **推荐使用场景** | 需要外部传递参数的模块（如可配置位宽） | **状态机状态编码**（不希望被外部修改） | 全局常量（如仿真周期定义），但易污染命名空间 |
| **调用方式** | 直接使用变量名（如 `S0`） | 直接使用变量名（如 `S0`） | 加反引号调用（如 `` `S0 ``） |
| **是否加分号** | 需要分号结尾 | 需要分号结尾 | **不加分号**（否则分号成为宏内容的一部分） |

### 5.2 状态机编码定义的最佳实践

- 状态机编码推荐用 **`localparam`** 定义，因为状态码通常不希望被外部模块修改，且作用域清晰。
- 若用 `parameter` 定义，需注意实例化时可能被上层无意中重定义，导致状态机行为异常。
- 若用 `` `define `` 定义，需注意不要加分号，且全局生效，可能与其他模块宏名冲突，故**不推荐**。

## 6. 有限状态机设计要点

### 6.1 起始状态的选择

- **起始状态**：复位后状态机进入的初始状态。
- 必须明确定义起始状态，否则复位后状态寄存器初值为 `x`（不定态），状态机无法正常工作。
- 多数综合工具允许用户指定起始状态（通过复位赋值或综合属性）。
- **实践建议**：在状态定义中显式指定一个状态为起始状态（如 `S0`），复位时将 `state` 赋值为该状态。

### 6.2 同步复位 vs 异步复位（状态机视角）

| 复位方式 | 敏感列表 | 复位生效时机 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- | :--- |
| **同步复位** | 仅含 `posedge clk` | 仅在时钟沿检测到复位信号时生效 | 利于时序收敛，复位信号毛刺不易影响电路 | 复位信号必须持续到时钟沿到来 |
| **异步复位** | 含 `posedge clk or posedge clr` | 复位信号**立即生效**，不依赖时钟 | 响应快，无需时钟即可复位 | 复位信号毛刺可能导致误复位，时序分析复杂 |

### 6.3 多余状态（无效状态）的处理 

**问题来源**：当状态数不是2的整数次幂时（如5个状态用3位编码，存在3个未使用的码字），若电路受干扰进入无效状态，必须保证能**自主恢复**，否则状态机“死锁”。

**两种处理方法**：

| 方法 | 实现方式 | 效果 |
| :--- | :--- | :--- |
| **`default` 分支** | 在次态 `case` 语句末尾添加 `default: next_state <= S0;` | 任何未列出的状态码都导向起始状态 |
| **显式定义** | 在 `case` 中列出所有状态码，包括无效状态 | 更精确控制每个无效状态的下一跳 |

> **注意**：若 `case` 语句缺少 `default` 分支，且未列出所有状态，综合器会推断出 **锁存器（Latch）** 或导致不可预测行为。

**写出完整次态逻辑时，必须考虑的三件事**：
1. 所有输入组合下都有确定的下一状态（次态函数完备）。
2. 所有状态（包括无效状态）都有确定的下一状态。
3. 所有输出在所有状态下都有确定的赋值（避免输出锁存器）。

## 7. 如何在一个always块中给出一个完整的有限状态处理过程（单过程写法要点）

当使用**单过程描述**时，需要在同一个 `always @(posedge clk or posedge clr)` 内同时完成三件事。操作要点如下：

1. **复位分支优先**：在块开头判断复位信号，若有效则将状态寄存器赋为起始状态，输出赋为初始值，并提前退出（`if` 分支结束）。

2. **状态寄存器采用非阻塞赋值**：`state <= next_state` 是时序逻辑，必须用 `<=`。

3. **输出有两种处理方式**：
   - **方式A**：输出使用**阻塞赋值 `=`**，与状态更新语句并行存在，此时输出是**组合逻辑**产生的，紧跟当前状态。但需注意在单过程时序块中混合 `=` 和 `<=` 可能引起仿真与综合不一致，多数规范的单过程写法**仅用 `<=`**，输出会晚一拍。
   - **方式B（纯时序）**：所有赋值都用 `<=`，输出如同寄存器，在时钟沿更新，会**晚一拍**（一个时钟周期延迟），需要根据设计规格确认是否符合时序要求。

4. **`case` 结构完整**：`case(state)` 中每个状态都需对**所有输入组合**给出明确的下一状态；`default` 分支必须写，覆盖无效状态。

5. **输出赋值放在每个分支内部**：每个状态分支内同时指定下一状态和输出值，确保任何情况下输出都有确定值，不产生锁存器。

> 单过程描述的优缺点：优点是代码紧凑；缺点是状态和输出耦合，不易维护，且输出晚一拍容易遗忘，容易造出前后仿不一致的电路。

# 第10章 设计的优化 

## 10.1 设计的可综合性

**综合**：将Verilog/VHDL描述的行为级或功能级电路模型，转化为RTL级功能块或门级电路网表的过程。

### 可综合设计要点

| 序号 | 要点 | 说明 |
| :--- | :--- | :--- |
| 1 | **不使用初始化语句** | 如 `reg q = 0;` 不可综合，复位应通过 `reset` 信号实现 |
| 2 | **不使用带延时的描述** | 如 `#5 a = b;` 中的 `#5` 不可综合，延时仅用于仿真 |
| 3 | **不使用循环次数不确定的循环语句** | `forever`、`while` 不可综合；`for` 的循环次数必须是常数 |
| 4 | **尽量采用同步方式设计** | 全局时钟+同步复位/使能为宜，便于时序收敛 |
| 5 | **组合逻辑敏感列表必须完整** | 使用 `always @(*)` 避免遗漏信号，防止生成锁存器 |
| 6 | **所有内部寄存器应能够被复位** | 使用全局复位端（`clr`/`reset`） |
| 7 | **任务（task）和函数（function）** | 通常被综合为**组合逻辑**模块 |

> **仿真语句与可综合语句的区分**：`initial`、`#延时`、`forever`、`while`、`repeat`（大部分）、`fork-join` 等均**不可综合**，仅用于测试平台（Testbench）。


## 10.2 流水线设计（Pipeline Design）技术 ⭐

### 10.2.1 流水线的基本原理

**目的**：提高系统最高工作频率（时钟频率）。

**核心思想**：将**长延时的组合逻辑路径**（关键路径）分解为多个**延时较短的步骤**，在每一步之间插入**寄存器（触发器）**暂存中间结果，使系统能工作在更高的时钟频率下。

**代价**：增加了寄存器数量（面积开销），同时数据流会产生**延迟（Latency）**。

### 10.2.2 非流水线 vs 流水线对比

| 对比项 | 非流水线 | 流水线 |
| :--- | :--- | :--- |
| **工作方式** | 所有逻辑在一个时钟周期内完成 | 拆分为多个步骤，每个步骤在一个时钟周期内完成 |
| **关键路径延时** | 长（T = 总组合逻辑延时） | 短（T ≈ 单级组合逻辑延时 + 寄存器延时） |
| **最高时钟频率** | 低（受限于总延时） | 高（受限于最慢的一级） |
| **输出延迟（Latency）** | 1个时钟周期 | N个时钟周期（N为级数） |
| **吞吐率（Throughput）** | 每个周期1个结果 | **每个周期仍为1个结果**（但首次输出需等待N周期） |
| **资源消耗** | 寄存器少 | 寄存器多（每级插入一组寄存器） |
| **适用场景** | 组合逻辑深度小 | 复杂算术运算（乘法、加法树）、高速接口 |

-  **结论**：流水线级数越多，单级延时越短，最高工作频率越高；但寄存器开销增大，输出延迟（Latency）也相应增加。

### 10.2.3 流水线的核心公式

- **非流水线**：`T_max = T_comb`（组合逻辑延时），`F_max = 1 / T_comb`
- **N级流水线**：`T_max ≈ T_comb / N + T_reg`（每级插入寄存器），`F_max ≈ 1 / (T_comb/N + T_reg)`
- **吞吐率**：流水线不改变每个时钟周期产出1个结果的吞吐率，但**首次输出需要N个周期的初始化延迟（Latency）**。


## 10.3 资源共享（Resource Sharing） ⭐

### 10.3.1 资源共享的核心思想

**目的**：减少硬件资源消耗（尤其是面积较大的模块，如乘法器、加法器、RAM等）。

**核心思想**：用多路选择器（MUX）将多个数据源分时接入同一个功能模块（如加法器），替代多个功能模块并行工作。

### 10.3.2 两种实现方式对比（PPT案例）

**场景**：根据选择信号 `sel`，计算 `a+b` 或 `c+d`。

| 对比项 | 方式一（资源不共享） | 方式二（资源共享） |
| :--- | :--- | :--- |
| **结构** | 2个加法器并行，由MUX选择输出 | 1个MUX选择数据源，送入1个加法器 |
| **资源消耗** | **2个加法器** | **1个加法器 + 1个MUX** |
| **延时** | 较短（加法器与MUX串行） | 较长（MUX + 加法器串行延时） |
| **适用场景** | 对速度要求高 | 对面积/成本要求高，速度可降低 |

> **关键结论**：硬件代价高的功能模块（加法器、乘法器）通过增加少量MUX进行共享，**以速度换面积**，可显著降低系统资源消耗。

## 10.5 多层次结构电路的设计

### 10.5.1 Top-down 设计方法

- 将复杂系统逐级分解为模块 → 子模块 → ... → 可独立实现的简单单元。
- 便于分工协作，类似软件工程中的模块化设计。

### 10.5.2 模块调用的两种方式

| 调用方式 | 语法格式 | 特点 |
| :--- | :--- | :--- |
| **位置对应** | `模块名 例化名 (信号1, 信号2, ...);` | 端口顺序必须与模块定义完全一致，易出错 |
| **信号名对应** | `模块名 例化名 (.端口名(连接信号), ...);` | 端口顺序无关，**可读性好，工程推荐** |

**示例**：
```verilog
// 位置对应（不推荐）
reg8 accreg8(accout, sum, clk, clear);

// 信号名对应（推荐）
reg8 accreg8(.qout(accout), .in(sum), .clk(clk), .clear(clear));
```


## 10.6 进程（Process）

### 10.6.1 进程的概念

- 行为模型本质上是**进程**的集合。一个进程是一个独立的运行单元。
- 数字系统的行为可看作多个进程的有机结合。

### 10.6.2 Verilog中的进程形式

| 进程形式 | 说明 |
| :--- | :--- |
| `always` 过程块 | **反复执行**，描述时序逻辑或组合逻辑 |
| `initial` 过程块 | **只执行一次**，仅用于仿真初始化 |
| `assign` 持续赋值语句 | 等价于一个**组合逻辑进程**，输入变化即重新计算 |
| 元件调用（门级实例化） | 等价于一个**硬件进程**，输入变化驱动输出 |

### 10.6.3 进程设计注意事项

| 序号 | 要点 |
| :--- | :--- |
| 1 | 组合逻辑和时序逻辑应**分配到不同的进程**（`always`块）中 |
| 2 | 一个进程中**只允许描述对应于一个时钟信号**的同步时序逻辑 |
| 3 | 进程必须由敏感信号变化启动，**敏感列表必须精心选择** |
| 4 | 多个进程之间通过**信号（wire/reg）**进行通信，可设置**握手信号**协调运行 |


## 10.7 阻塞赋值与非阻塞赋值

本章内容与第6章完全一致，此处仅列出要点作为复习索引：

| 序号 | 原则 |
| :--- | :--- |
| 1 | 组合逻辑（`always @(*)`）用**阻塞赋值 `=`** |
| 2 | 时序逻辑（`always @(posedge clk)`）用**非阻塞赋值 `<=`** |
| 3 | 锁存器建模用非阻塞赋值 |
| 4 | 同一 `always` 块中同时含组合和时序逻辑时，推荐非阻塞赋值 |
| 5 | 禁止在同一 `always` 块中混合使用 `=` 和 `<=` |
| 6 | 严禁在两个或多个 `always` 块中对同一变量赋值（多驱动冲突） |


## 10.8 FPGA设计中毛刺的消除 ⭐

### 10.8.1 毛刺（Glitch）的产生原因

- **根本原因**：信号在FPGA内部的连线和逻辑门中传输存在**传播延时**。
- **表现形式**：多路信号电平变化时，由于各路径延时不同，组合逻辑输出端在稳定之前会出现**不期望的尖峰脉冲**。
- **为何必须处理**：PCB走线有分布电感和电容，能自然滤除部分毛刺；但**FPGA内部无分布电感和电容**，毛刺会完整保留并向后级传递，可能导致系统误动作。

> **教材术语关联**：毛刺现象在数字电路中称为 **“冒险”（Hazard）** ，包括逻辑冒险和功能冒险。

### 10.8.2 毛刺的消除方法

| 方法 | 原理 | 示例 |
| :--- | :--- | :--- |
| **改变编码方式** | 减少状态转移时的位翻转数，破坏毛刺产生的条件 | 用**格雷码计数器**替代二进制计数器（相邻状态仅1位变化） |
| **引入D触发器（输出寄存）** | D触发器对输入端的毛刺**不敏感**（只有在时钟沿采样），将组合逻辑输出寄存一拍后再输出 | 在组合逻辑输出后加一级触发器（见PPT例10.23→10.24） |

### 10.8.3 核心案例：长帧同步时钟的毛刺消除

**问题代码（产生毛刺）**：
- 组合逻辑 `always @(counter)` 根据计数器值直接产生 `strb` 输出。
- 当 `counter` 多位同时变化时，`strb` 在稳定前出现毛刺。

**改进方法（消除毛刺）**：
- 在组合逻辑输出 `temp` 后，增加一级 **`always @(posedge clk)`**，将 `strb <= temp` 寄存输出。
- 组合逻辑的毛刺在触发器D端出现，但**只有在时钟边沿才被采样**，而毛刺宽度通常小于时钟周期，不会被采到，从而输出干净。

> **核心原理**：触发器（寄存器）对D端的毛刺不敏感，因为D端信号必须满足**建立时间（Setup Time）和保持时间（Hold Time）**要求才会被正确采样，窄脉冲毛刺不满足建立时间要求，因此被有效滤除。
